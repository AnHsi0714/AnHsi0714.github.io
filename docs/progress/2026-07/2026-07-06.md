### 朋友創作：修「下一個作品」按鈕點了沒反應

使用者回報 `/friends` 創作牆點「›」下一個作品常常沒反應，一路排查換了三版才抓到真正原因：

- **第一輪誤判**：先猜是 `centerItem` 用的 `el.scrollIntoView({ behavior: "smooth" })` 跟 scrollArea 的 `scroll-snap-type`（`snap-x snap-proximity`）衝突——部分瀏覽器（尤其 Chromium／Windows）smooth 捲動途中會被 snap 邏輯拉回原位。改成自己算目標位置、對 container 呼叫 `scrollTo`，一樣用 smooth；使用者回報「還是有時有效有時沒效」。
- **第二輪誤判**：改用 `behavior: "instant"`（`GalleryGrid.tsx` 的展牆捲動用同一招解過類似問題），同時猜測是 React Query 背景重新請求資料（預設 `staleTime: 0` + `refetchOnWindowFocus: true`，切視窗焦點就會重打 API）觸發了「捲動位置重置」的 effect（`useEffect(() => { scrollRef.current?.scrollTo({left:0}); ... }, [creations, ...])`），把該 effect 的依賴改成只在掛載時跑一次；另外 `fetchFriendCreations()` 的 `order("created_at")` 沒有唯一鍵，同秒建立的兩筆作品在重新查詢時排序可能不穩定，一併加上 `order("id")` 當次要排序鍵保證順序固定。還是一樣，而且只有改完 code 那次會動，後續按都沒反應。
- **真正原因**（靠加 console.log 貼出實際數值才抓到，純猜測猜了兩輪都不對）：`FriendCreationCarousel` 原本用 `itemRefs = useRef<(HTMLDivElement|null)[]>([])` 手動記錄每張卡片的 DOM 節點，每次重繪在 render phase 把它清空（`itemRefs.current = []`），指望 ref callback 在 commit 階段重新填回去。實測 log 顯示連續點擊時 `itemRefsLength=0`、`hasItem=false`——代表某次重繪把陣列清空後，ref callback 沒有可靠地在下次點擊前把它填回去，`centerItem` 於是直接判定「這個位置沒有元素」而 no-op，完全沒反應。
- **修法**：拿掉 `itemRefs` 陣列，改成每次要用時直接 `scrollRef.current.querySelectorAll("[data-index]")` 現查 DOM（卡片改標 `data-index={index}` 取代 ref callback），不依賴 ref 掛回去的時機。這正是 `GalleryGrid.tsx` 展牆捲動已經在用、驗證穩定的作法，兩個 carousel 現在用同一套模式。
- **教訓**：這類「捲動＋大量互動狀態」的 bug 沒有瀏覽器工具幾乎不可能純看程式碼猜對，前兩輪修正方向（scroll-snap、資料重新請求）雖然都是合理、也確實有修到的小問題（排序穩定性、不必要的捲動重置），但都不是本因；真正定位靠的是請使用者貼實際 console.log 數值，而不是繼續憑理論排查。
