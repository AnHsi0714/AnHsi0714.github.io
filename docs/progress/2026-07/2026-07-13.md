### 全站雙語（zh/en）i18n 架構上線

在此之前全站沒有語言切換，所有內容與 UI 文案都寫死繁中。這次一次補齊：

- **切換機制**：NavBar 上的 `LanguageToggle` 按鈕，狀態由 `LanguageContext`（`LanguageProvider` / `useLanguage()`）管理，存進 `localStorage`（key `language`）；不走 URL 語言前綴（`/en/...`），中英文共用同一套 SPA 路由。
- **UI 文案**：`src/i18n/strings.ts` 定義 `Strings` 介面＋`strings.zh`／`strings.en` 兩份物件；`useTranslation()` 回傳 `{ t, language }`，`t` 是整包文案物件（呼叫端用 `t.nav.about`，不是 key-lookup 函式）。
- **內容翻譯**：走「同名 `.en.` 分身檔」慣例（`foo.json` → `foo.en.json`、`foo.md` → `foo.en.md`），不做同檔雙語欄位——中文內容維持原樣不動，英文另外整份寫。適用於 `content/*.json`（projects／dreams／experience／artworks／glossary）與 `content/{articles,projects}/*.md`。Markdown 讀取（`src/lib/articles.ts`、`src/lib/projects.ts`）同時 glob `*.md` 與 `*.en.md`，缺 `.en.md` 分身時 fallback 回中文版；JSON 內容改用 `useLocalized(zhData, enData)`（`src/lib/localized.ts`）在呼叫端切換，兩種語言都是靜態 import，不做 glob。
- **Glossary 名詞解釋也雙語化**：`content/glossary.json`／`.en.json`，`useGlossary()`（`src/lib/glossary.ts`）依語言取定義，`Term.tsx` 顯示對應語言。
- 同時補上 **SilkRoad**、**Super Mario** 兩個先前只有規劃、還沒寫文件的專案頁（含中英文版）。
- `/dev/*`（ComponentsPreview／CreatureDemo／CreatureBuilder）刻意不做翻譯：屬於內部拋棄式開發工具，不是對外 portfolio 內容，排除在 i18n 範圍外。

決策依據：使用者要求中文內容完全不動、英文頁面要有完整翻譯，在「URL 語言前綴路由」與「同檔雙語欄位」兩個方案之間選了「toggle + Context + 同名 `.en.` 分身檔」。

### 站內導覽捲動位置修正（「查看完整經歷」跳到頁面中段）

`App.tsx` 原本的捲動 effect 只處理有 hash 的情況（`if (!hash) return`），沒有 hash 時完全不動作。React Router 換頁不像整頁重新載入會自動把捲軸歸零，從 About 頁面下方點「查看完整經歷」連結到 `/experience` 時，會直接沿用 About 頁面當下的捲動位置，畫面停在新頁面中段甚至底部，而不是從頂端開始讀。改成沒有 hash 時明確呼叫 `window.scrollTo(0, 0)`，讓所有站內導覽（不只這個連結）換頁後都固定回到頁首，只有明確帶 hash 的連結才維持原本的 `scrollIntoView` 行為。

### CTA 按鈕主次對比、畫廊改名

- **按鈕主次視覺層級**：`Button.module.scss` 的 `.primary`／`.secondary` 原本都是透明底＋細邊框，只靠文字顏色微妙區分，兩顆按鈕放在一起（如首頁「關於我」／「查看專案」）幾乎看不出主次。中間試過兩種路線都被打回：①把 `.primary` 邊框加粗到 2~3px 並改用黑色文字／邊框——使用者測試後覺得填色配透明的組合「背景不適合」，退回原本的主色系文字；②把邊框加粗直接搬進 `.btn` 基底類別讓全部 variant 統一變粗——導致工具列裡 ghost／danger 按鈕高度被連帶撐高，也不是使用者要的「只調整 primary」。最後定案：`.primary` 邊框維持跟其他 variant 一樣的 1px（高度統一，不用另外幫其他 variant 補 padding），改用 `box-shadow: inset 0 0 0 2px` 疊一圈視覺上的粗框——box-shadow 不佔版面空間，純視覺加粗，不會影響任何按鈕的實際高度。
- **畫廊改名**：中文「藝術畫廊」→「生成視覺」（維持 4 字，跟原長度一致），英文「Gallery」→「Generative Visuals」，NavBar、首頁「更多內容」卡片、`/gallery` 頁面自己的標題三處同步更新（`strings.ts`）。

`tsc --noEmit`、`vite build` 皆通過。

### NavBar 從 fixed 改 sticky，修正手機版標題蓋住 bug

調整首頁排版時，實機截圖發現窄螢幕下 NavBar 換行變成 3 行，但 `App.tsx` 的 `<main>` 只用固定的 `pt-20` 預留單行導覽列高度，導致語言／主題切換鈕蓋住首頁標題文字（「鄭安瑞 Cheng An Hsi」只剩「Hsi」露出來）。根本原因是 NavBar 用 `fixed` 定位、不佔文件流高度，`pt-20` 只是憑經驗猜的補償值，換行行數一多就對不上。改成 `sticky top-0`，讓 NavBar 佔用真實版面高度、`<main>` 移除 `pt-20`——不管換幾行都不會蓋到內容，全站頁面（不只首頁）都受惠。

### 首頁 hero 拿掉 vw 出血 margin，改用一般容器留白

`Home.tsx` hero 區塊原本用 `lg:ml-[calc((min(100vw,64rem)-100vw)/2-1rem)]` 讓標題在寬螢幕出血到畫面左緣。這個算式依賴 `100vw`，但 `100vw` 在有實體捲軸的瀏覽器裡不含捲軸寬度（`index.css` 裡其實已有相關註解點出這個坑）；在大字級（`text-7xl`）下這個誤差被放大，使用者截圖顯示標題被裁到只剩「Hsi」，中間好幾個字元直接消失在畫面外。用 headless Chromium 測試時因為是覆蓋式捲軸、`100vw` 沒有誤差，沒能重現，只有實機瀏覽器才踩得到。直接拿掉這個出血 margin，hero 改用跟其他區塊一樣的一般置中容器留白；同時把 hero 內部元素間距加大（`mt-2→mt-3`、`mt-4→mt-5/mt-6`、`mt-6→mt-8`）、補上 `md:text-6xl` 讓平板寬度的字級不會從 `sm` 直接跳到 `lg`。

### 「生成視覺」畫廊手機版：橫向展牆＋吊燈 → 直向清單＋環境光暈

畫廊原本的桌機設計是橫向捲動的展牆，每件作品上方掛一盞吊燈（燈繩／燈罩／燈泡）打光束下來，捲到哪盞燈就亮到哪。這個橫向捲動＋吊燈的隱喻在觸控裝置上不好操作（跟頁面手勢衝突），使用者要求手機版拿掉橫向移動跟燈本體，改成直向捲動、光效改成從畫作四周擴散的光暈。

- `GalleryGrid.module.scss` 新增 `@media (max-width: 640px)`：`.scroller` 改 `flex-direction: column` 直向捲動；`.lamp`／`.beam` 直接隱藏；`.frame` 用 `::before` 徑向漸層＋模糊做環境光暈，亮度沿用原本的 `--lit` 變數（跟著目前置中的作品連動）。
- 新增 `src/hooks/useMediaQuery.ts`，`GalleryGrid.tsx` 用 640px 閾值判斷手機版，把「哪件作品在中央」的偵測、`scrollIntoView` 的置中方向從水平軸切成垂直軸；滾輪轉橫向捲動的攔截邏輯手機版直接跳過（觸控本來就沒有滾輪事件，不該攔）。
- 標題區塊原本用 `position: absolute` 浮在展牆上層（桌機橫向捲動不會把畫作捲到標題底下），手機版改直向捲動後會互相蓋住，改回正常版面流接在標題下面。
- 說明文字「左右捲動瀏覽畫廊」改成方向中立的「捲動瀏覽畫廊」（中英文皆改）。

### 畫廊房間垂直捲軸溢出頁面（NavBar 改 sticky 的連鎖影響）

使用者回報桌機版「生成視覺」出現垂直捲軸、稍微溢出頁面。追出來是同一批 NavBar fixed→sticky 改動的連鎖效應：`.room` 原本用 `margin-top: -5rem` + `height: 100vh` 這組數字，是配合舊版「NavBar `fixed` 不佔版面、`main` 用 `pt-20` 手動避開」算出來的；NavBar 改 sticky 後這組寫死的數字對不上，多出約 8px 高度，逼出頁面級捲軸。

這次順便把它改成不會再過時的動態算法，而不是重新猜一組新的寫死數字：`NavBar.tsx` 用 `ResizeObserver` 即時量出導覽列自己的實際高度，寫入 CSS 變數 `--nav-h`（NavBar 有 `flex-wrap`，換行時淨空本來就會變，手機版最多疊到 3 行）；`.room` 的高度改成 `calc(100dvh - var(--nav-h))`，`margin-top` 改成精確抵銷 `main` 自己的 `pt-8`（`-2rem`），讓房間貼齊 NavBar 底部、不留縫也不溢出；`.header` 的 `top` 跟著等比例從 `5rem` 調成 `2rem`。用 Playwright 量測 1440／1024／768／390 四種寬度，`scrollHeight` 都精確等於 `innerHeight`（差值 0px），`--nav-h` 也隨 NavBar 換行正確變化（62px／102px／138px）。

順手查了一下，作品詳細頁 `GalleryDetail.module.scss` 的 `.stage` 也是同一套「配合舊版 fixed NavBar」的 `margin-top: -5rem`／`padding-top: 5rem`／`min-height: 100vh` 寫法，同樣的 8px 溢出問題還沒被回報就先一併修掉：`margin-top` 改 `-2rem`（精確抵銷 `main` 的 `pt-8`）、`padding-top` 等比例調成 `2rem`、`min-height` 改用 `calc(100dvh - var(--nav-h))`；`.back` 返回連結的 `top` 位移量跟著同步從 `5rem` 改成 `2rem`。
