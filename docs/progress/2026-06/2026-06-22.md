- 排查並修好 GitHub Pages 部署失敗（瀏覽器報 `application/octet-stream` MIME 錯誤）：
  - 根因：`.github/workflows/deploy.yml` 的 push trigger 寫的是 `master`，但 repo 的分支其實是 `main`，build+deploy 的 workflow 從未真正跑過；GitHub Pages 因此直接 serve 沒編譯過的 `index.html`，裡面的 `<script src="/src/main.tsx">` 指向原始 `.tsx` 檔，瀏覽器拿到的 MIME type 自然不對。
  - 修法：trigger 改成 `branches: [main]`。
- 排查並修好 `npm ci` 在 CI 上失敗（`package-lock.json` 與 `package.json` 不同步）：
  - 第一階段：`package.json` 有 `@fortawesome/*` 套件但 lock file 沒同步更新，補跑 `npm install` 重新產生。
  - 第二階段：修完 fontawesome 後，CI 仍報 `Missing: @emnapi/core@1.11.1`。一路排查發現真正原因是本機 Windows 的 npm（11.x）跟 GitHub Actions 用的 npm（10.9.8，隨 `actions/setup-node` 的 `node-version: 22` 綁定）對同一個 optional 依賴（`@tailwindcss/oxide-wasm32-wasi` 宣告的 `@emnapi/core: ^1.10.0`）解析結果不同：npm 11 會把它去重成另一個套件精確釵住的 `1.10.0`，npm 10.9.8 卻會提升成更新的 `1.11.1`，所以在本機（不管怎麼清 cache、清 node_modules）重新產生的 lock file 永遠無法滿足 CI。
  - 修法：用 Docker 跑 `node:22` 容器（裡面的 npm 版本跟 CI 一致），在那個容器內重新產生 `package-lock.json`，並在同一個容器裡驗證 `npm ci` + `npm run build` 都過，才 commit。
  - 過程中有一版「本機 Windows 產生的乾淨 lock file」先被 merge 進 main，後來才確認那版其實還是錯的；發現真正原因後，新分支跟 main 上已 merge 的舊修正在 `package-lock.json` 上互相衝突，用 rebase 解掉，保留 Docker 驗證過的版本。
  - 永久修法：在 `deploy.yml` 的 `npm ci` 前面加一步 `npm install -g npm@11.6.2`，把 CI 用的 npm 版本釵死成跟本機一樣，不再讓它隨 `node-version: 22` 飄動到「當下最新的 22.x 附帶版本」。同步把本機 lock file 用 npm 11.6.2 重新產生一次，並在跑過同樣 pin 流程的 Docker 容器內驗證 `npm ci` + `npm run build` 都過。
  - 更完整的觀察：這次落差不是單純「npm 10.9.8 vs 11.x」就能完全解釋，`package.json` 裡 `"vite": "^8.0.16"` 是浮動範圍，期間 vite 從 8.0.16 升到 8.1.0、連帶 `rolldown` 從 1.0.3 升到 1.1.3、它精確釵住的 `emnapi` 版本也跟著變——所以即使 npm 版本釵住了，只要日後又整個刪掉 `node_modules`／`package-lock.json` 重新 `npm install`，一樣有機會因為上游又出新版而再漂移一次。之後要新增套件，建議只用 `npm install <套件>` 做最小變更，避免整個重新解析；真的要整個重建 lock file 時，才需要重新跑一次 Docker 驗證。
